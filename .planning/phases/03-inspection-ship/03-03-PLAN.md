Credit formula: actual_credits = (prompt_tokens + completion_tokens) × multiplier

SELECT FOR UPDATE concurrency model (from PROJECT.md):
  T=0  Request arrives → BEGIN TX → SELECT user FOR UPDATE
  T=0  Check: used_credits + reserved_credits + estimated_credits ≤ quota  → else 402
  T=0  reserved_credits += estimated_credits → COMMIT (lock released)
  T=1  AI layer generates → returns actual token counts (outside any TX)
  T=2  BEGIN TX → SELECT user FOR UPDATE
  T=2  reserved_credits -= estimated_credits
  T=2  used_credits += actual_credits → COMMIT
  T=2  Write usage_log row

Alice example (from PROJECT.md — reproduce verbatim):
  User: alice, quota=1000, multiplier=2.0, used=800
  First request: prompt="explain recursion" (44 chars → ~11 tokens)
  estimated_completion=150, estimated_total=161, estimated_credits=322
  Check: 800 + 322 = 1122 > 1000 → 402
  (Alice pays, quota raised to 1200)
  Second attempt: Check: 800 + 322 = 1122 ≤ 1200 → PASS
  Reserve 322; generate → actual prompt=11, completion=178, total=189
  actual_credits = 189 × 2.0 = 378
  Settle: used = 800 + 378 = 1178, reserved = 0, remaining = 22

Known limitation (from Phase 2, D-17):
  If the settle transaction fails (DB down between T=1 and T=2), AI has generated text
  but credits are not debited — the user gets generation without being charged (undercharge).
  The reservation is released in the finally block. This is documented, not fixed, as the
  probability is low and the impact is bounded to one request.

Provider abstraction:
  BaseProvider ABC: estimate_tokens(prompt) → int, generate(prompt) → GenerationResult
  MockProvider: default (chars/4 heuristic, random ±10% completion variance)
  ClaudeProvider: opt-in via USE_REAL_LLM=true + ANTHROPIC_API_KEY; uses Anthropic API

Explain that the SELECT FOR UPDATE lock is held only during the lightweight check+reserve (not during generation), so concurrent requests queue at the DB lock without blocking AI generation.

4. **Alice Example** — Reproduce the full concrete quota example from PROJECT.md verbatim (1000-credit quota, multiplier=2.0, 44-char prompt, 402 then quota raise to 1200 then success, settle to used=1178 remaining=22).

5. **Credit Formula** — `actual_credits = (prompt_tokens + completion_tokens) × multiplier`. Explain estimate uses the same formula with estimated tokens (chars/4 heuristic or Anthropic count_tokens API). Actual differs because completion tokens are not known until after generation — this is expected and logged per row.

6. **Provider Abstraction** — BaseProvider ABC with estimate_tokens and generate methods. MockProvider default (no API key needed, deterministic for testing, ±10% completion variance simulates real estimate/actual gap). ClaudeProvider opt-in via USE_REAL_LLM=true environment variable.

7. **Key Tradeoffs** — Must address:
   - **402 vs 429**: 402 ("Payment Required") is semantically correct for quota exhaustion — the user owes more credits. 429 means rate-limited by time, not by budget. Using 402 communicates the right signal to callers.
   - **Debit actual, not estimate**: Fairest to the user. Overrun is possible if actual > estimate, but bounded to at most one request's worth. Documented in QUOTA-05.
   - **Lock duration**: Lock held only during reserve TX (~1ms DB roundtrip), not during AI generation (100-2000ms). This maximizes throughput while preserving correctness.
   - **No Redis**: Postgres row lock is sufficient. Adding Redis would require synchronizing two data stores — more complexity for the same guarantee.

8. **Known Limitation** — If the settle transaction fails after AI has generated (DB crash between T=1 and T=2): AI returns text to the user but credits are not debited (undercharge). The reservation is released in the finally block so the user's available credits are restored. Probability is low; impact is bounded to one request. A production system would use outbox pattern or idempotent retry.
